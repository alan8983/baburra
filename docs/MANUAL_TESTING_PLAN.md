## 一、測試環境與前提條件

### 1.1 環境配置要求

| 項目 | 要求 |
|------|------|
| Node.js | >= 18.x |
| 瀏覽器 | Chrome 120+ / Edge 120+ / Firefox 120+ |
| 開發伺服器 | `npm run dev` 啟動於 `http://localhost:3000` |
| Supabase | 本地或雲端 Supabase 專案已連結 |

### 1.2 環境變數確認

確保 `.env.local` 包含以下變數：

```bash
# 必要
NEXT_PUBLIC_SUPABASE_URL=<your-supabase-url>
NEXT_PUBLIC_SUPABASE_ANON_KEY=<your-anon-key>
SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>

# AI 功能測試需要
GEMINI_API_KEY=<your-gemini-api-key>

# K 線圖功能測試需要
TIINGO_API_TOKEN=<your-tiingo-token>
```

### 1.3 資料庫初始化選項

| 測試情境 | 執行指令 | 說明 |
|----------|----------|------|
| **含種子資料** | `supabase db reset` | 包含 KOL、股票、文章等假資料 |
| **空白環境** | `supabase db reset --no-seed` 後執行 `supabase db execute -f supabase/seed-minimal.sql` | 僅論點類別，無假內容 |

> 詳細設定請參考 [MANUAL_TESTING_SETUP.md](./MANUAL_TESTING_SETUP.md)

### 1.4 測試帳號

| 帳號類型 | Email | 密碼 | 用途 |
|----------|-------|------|------|
| 測試帳號 A | `test-a@example.com` | `Test1234!` | 主要測試用 |
| 測試帳號 B | `test-b@example.com` | `Test1234!` | RLS 隔離測試 |

> **注意**: 若使用空白環境，需先註冊帳號

## 二、測試案例格式說明

### 2.1 測試案例編號規則

```
TC-[Phase編號]-[流水號]

範例：TC-04-001 = Phase 4 的第 1 個測試案例
```

### 2.2 嚴重程度定義

| 等級 | 標籤 | 定義 |
|------|------|------|
| P0 | Critical | 系統無法使用、資料遺失、安全漏洞 |
| P1 | High | 主要功能無法正常運作 |
| P2 | Medium | 次要功能異常，有替代方案 |
| P3 | Low | UI 瑕疵、文案錯誤、建議改善 |

### 2.3 測試狀態

| 狀態 | 說明 |
|------|------|
| ⬜ Not Started | 尚未執行 |
| 🔄 In Progress | 執行中 |
| ✅ Passed | 通過 |
| ❌ Failed | 失敗 |
| ⏭️ Skipped | 跳過（前置條件不符） |
| 🚧 Blocked | 被阻擋（依賴其他問題） |


**前置條件**:

| 3 | 手動計算驗證 | - | 從 Supabase 取得文章資料並手動計算 | - |

**計算邏輯驗證**:
```
勝率 = 勝場數 / 總有效場數 × 100%

勝場判定：
- 情緒 > 0 (看多) 且 漲跌幅 > 0 → 勝
- 情緒 < 0 (看空) 且 漲跌幅 < 0 → 勝
- 情緒 = 0 (中立) → 不計入
```

**Supabase 驗證**:
```sql
-- 取得 KOL 所有文章的情緒和漲跌幅
SELECT p.id, p.sentiment, p.posted_at, s.ticker
FROM posts p
JOIN post_stocks ps ON p.id = ps.post_id
JOIN stocks s ON ps.stock_id = s.id
WHERE p.kol_id = '<kol_id>' AND p.status = 'Published';
```

**通過標準**:
